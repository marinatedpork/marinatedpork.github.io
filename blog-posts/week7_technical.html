SQL Injection

August 28, 2014

What they are and how we avoid them

SQL Injection is a code injection technique in which SQL statements are inserted into an entry field so that it can be executed. These injections are by and large malicious attacks by people trying to disrupt your system. The results of which can be very costly when you have someone enter “DROP table tweets” while filling out a form. A server that is not equipped to handle such a threat takes the field entry as code and, in the case of entering “DROP tables tweets”, would thus delete your entire tweets table.


Obviously one wonders why some jerk would ever come along and disrupt your database like that. People have their motives, though. If you’re running a business with a competitor, they might try to set you back by disrupting your data. Otherwise, there might be someone out there who just wants to watch the world burn. Regardless, developers ought to implement means of preventing this.


You may avoid these SQL injections by being mindful of the following things:


  • Filter for literal escape characters (symbols or words that invoke interpretation) and also make sure your system is converting character types during interpretation or compiling. Each language may have a different procedure for doing this. Regardless the idea is to put a control in place that would, for example, turn all fields temporarily into strings and require a specific method for sending literal SQL commands.

  • Understand your languages type strength. Ruby for example is strongly and dynamically typed. This means that it a language where the type of a variable can be altered at any time and it is strict about how you can use your variable. Ask yourself, how is the user input going to be interpreted? What datatype is it?

  • Mitigate injection attacks at the code level with object relational mapping. Object-relational mapping (ORM) is essentially a means of creating a virtual object database than can be used within your programming language. In Ruby this would mean mapping sqlite3 or something similar into objects.

  • Implement parameterized statements by preparing statements with placeholders than can only store a certain type of information and not an arbitrary SQL fragment. In this case, the an SQL fragment would be treated as simply a very strange looking string.

  • Database permissions that deny certain literals or queries can also be implemented to simply blog users from selections and tables.

  • Create pattern checks that looks for specific values series in strings before passing along any information. For example, a string may be checked to for a date type or alphanumeric pattern and only pass it along if it matches the pattern you’ve implemented.
